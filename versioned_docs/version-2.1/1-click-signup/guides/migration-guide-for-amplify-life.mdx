:wave: **Hello Amplify Life team!** You're currently using 1-Click Signup with v2.0 of the Verified API, which we're soon deprecating. You'll need to migrate to v2.1, but don't worry: this is very easy to do.

**You need to make one decision: For phone verification, do you want to use our 1-Click Verify API or another solution?** For context: Currently, in v2.0, we're generating the verification code for you, and you're sending an SMS containing that verification code to the user. In v2.1, this is not an option. Instead, you have two choices:
1. **Use our 1-Click Verify API:** This is our dedicated phone verification solution, which didn't exist when you first integrated 1-Click Signup. With 1-Click Verify, we generate the verification code _and_ send the SMS for you. 
2. **Use another phone verification solution:** There are plenty of other dedicated phone verification solutions out there, such as Twilio Verify. Or, you can implement your own solution by generating verification codes (and still sending) SMS yourself.

:::info Why We Made This Change
By requiring a dedicated phone verification solution in v2.1 — whether our 1-Click Verify API or another solution — we enable an easier integration for you (because you don't have to implement SMS sending directly) and better security (because the verification code is _only_ exposed to the user).
:::

:::tip We Recommend 1-Click Verify
We recommend using our 1-Click Verify API, because it ties directly into 1-Click Signup to make the integration especially easy and secure. Specifically:
- 1-Click Verify outputs a `verificationUuid`.
- You can pass this `verificationUuid` directly to 1-Click Signup, which can then hydrate the user's phone number and confirm that it was recently verified.

In other words, if you use 1-Click Verify there's a closed loop with security guarantees, If you use another solution, we can't have such security guarantees, so instead we have to do a manual security and compliance review of your integration to ensure you're handling phone verification properly. 
:::

    **All you need to do is the following.** It should take no more than a day.

    1. **Generate and check phone verification codes:** With API integration type in v2.1, Verified does not generate or check phone verification codes for you, so you'll need to do this yourself. You can also use a dedicated third party service (like [Twilio Verify](https://www.twilio.com/en-us/user-authentication-identity/verify)) for this if you prefer.
    2. **Update your API calls:**
        1. **Remove the first `POST /1-click` call** (step 2 under Semi-Hosted in the table below). _This call was to have Verified generate a phone verification code, which in API v2.1 you do yourself._
        2. **Replace the `GET /1-click` call** (step 5 under Semi-Hosted in the table below). Call `POST /1-click` instead. _You'll receive either data or an error. The same error handling you have in place for the `GET` call will work for the `POST` call._
        3. **Replace the `PATCH /1-click` call** (step 7 under Semi-Hosted in the table below). Call `POST /1-click` instead (with additional info from the user). _You'll receive either data or an error. The same error handling you have in place for the `PATCH` call will work for the `POST` call._

    **That's it!**

## Updates in v2.1

The most significant updates in [v2.1](/) are:
- **We introduced an [SDK](/guides/sdk-integration),** which is _10x faster_ to integrate. It's highly customizable and handles the [user experience](/guides/user-experience) end to end, including sending SMS to users for phone verification.
- **We revamped our suite of [test users](/test-users),** making it far easier to test all possible flows of 1-Click Signup. Crucially, the new test users all have phone numbers that pass validation checks, making them much simpler to use for both manual and automated testing.
- **We changed the available integration types:** The [integration types in v2.0](/v2.0/integration-guide#2-configure-brand-settings) are updated to [integration types in v2.1](/guides/setup#integration-type) as follows:
    - **Hosted** in v2.0 is replaced by **SDK** in v2.1
    - **Semi-Hosted** in v2.0 is no longer available in v2.1
    - **Non-Hosted** in v2.0 is equivalent to **API** in v2.0

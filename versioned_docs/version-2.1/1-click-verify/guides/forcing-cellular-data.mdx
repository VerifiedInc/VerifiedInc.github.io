The 1-Click Verify autofill channel works over the cellular network, so it's not available when a user's device is routing traffic over WiFi. Web apps can't do anything about this, because browsers don't let you choose a network interface. **But native mobile apps can: both iOS and Android provide APIs to force specific requests over cellular data, even while the device is connected to WiFi.**

This guide shows you how, on each platform. A typical use is determining the device's cellular IPv6 address for the `deviceIp` parameter in [step 1a of the API Integration guide](/1-click-verify/guides/api-integration#a-check-if-autofill-is-available): make the request to your IP echo endpoint over cellular, and you get the device's cellular IPv6 back, regardless of WiFi.

:::caution This Doesn't Apply to WebViews
These techniques work for requests your native code makes. They can't control requests made *inside* a WebView (by page JavaScript, for example): on iOS, WKWebView's networking runs in a separate system process that your app can't configure. If part of your flow runs in a WebView, make the cellular-dependent request natively and pass the result into your web layer.
:::


## iOS

On iOS, use `Network.framework`. Creating an `NWConnection` with `NWParameters.requiredInterfaceType = .cellular` tells the system this connection may only use the cellular interface, so it goes over cellular even when WiFi is connected and would normally win.

There's no way to do this with `URLSession`: `URLSessionConfiguration` only offers `allowsCellularAccess`, which *permits* cellular but never *forces* it over WiFi. Because `NWConnection` is a raw connection rather than an HTTP client, you construct the HTTP request and parse the response yourself. The implementation below handles that for a simple GET: it sends HTTP/1.0 (so the server won't reply with chunked encoding), checks the status code, and returns the response body.

Two behaviors are important for correctness:

1. **The `.waiting` state is not a failure.** It fires while there's no usable cellular path *yet*, which is normal when WiFi is active and the cellular radio has to wake up. The connection recovers to `.ready` on its own.
2. **A hard timeout bounds the whole operation.** A device with no usable cellular (airplane mode, no plan) would otherwise sit in `.waiting` forever.

```swift title="CellularRequest.swift"
import Network

enum CellularRequestError: Error {
    case invalidURL
    case httpsOnly
    case timedOut
    case badResponse
    case httpError(statusCode: Int)
}

func forceCellularRequest(urlString: String,
                          timeout: TimeInterval = 10,
                          completion: @escaping (Result<String, Error>) -> Void) {
    guard let url = URL(string: urlString), let host = url.host,
          let rawPort = UInt16(exactly: url.port ?? 443),
          let port = NWEndpoint.Port(rawValue: rawPort) else {
        completion(.failure(CellularRequestError.invalidURL))
        return
    }
    // This minimal client always negotiates TLS; plain http:// would need
    // different NWParameters.
    guard url.scheme == "https" else {
        completion(.failure(CellularRequestError.httpsOnly))
        return
    }

    let params = NWParameters(tls: NWProtocolTLS.Options(), tcp: NWProtocolTCP.Options())
    params.requiredInterfaceType = .cellular

    let queue = DispatchQueue(label: "cellular-request")
    let connection = NWConnection(host: NWEndpoint.Host(host), port: port, using: params)
    var responseData = Data()
    var finished = false

    // Single exit point: completion fires exactly once and the connection is
    // always cancelled. Everything runs on `queue`, so `finished` is safe.
    func finish(_ result: Result<String, Error>) {
        guard !finished else { return }
        finished = true
        connection.cancel()
        completion(result)
    }

    func parseResponse() -> Result<String, Error> {
        guard let headerEnd = responseData.range(of: Data("\r\n\r\n".utf8)),
              let statusLine = String(decoding: responseData[..<headerEnd.lowerBound], as: UTF8.self)
                  .components(separatedBy: "\r\n").first else {
            return .failure(CellularRequestError.badResponse)
        }
        let fields = statusLine.split(separator: " ")
        guard fields.count >= 2, let status = Int(fields[1]) else {
            return .failure(CellularRequestError.badResponse)
        }
        guard (200..<300).contains(status) else {
            return .failure(CellularRequestError.httpError(statusCode: status))
        }
        return .success(String(decoding: responseData[headerEnd.upperBound...], as: UTF8.self))
    }

    func receiveLoop() {
        connection.receive(minimumIncompleteLength: 1, maximumLength: 65536) { data, _, isComplete, error in
            if let data = data { responseData.append(data) }
            if isComplete {
                finish(parseResponse())
            } else if let error = error {
                finish(.failure(error))
            } else {
                receiveLoop()
            }
        }
    }

    connection.stateUpdateHandler = { state in
        switch state {
        case .ready:
            let path = url.path.isEmpty ? "/" : url.path
            let query = url.query.map { "?\($0)" } ?? ""
            // HTTP/1.0 so the server won't respond with chunked encoding,
            // which keeps parseResponse() above valid.
            let httpRequest = "GET \(path + query) HTTP/1.0\r\nHost: \(host)\r\nConnection: close\r\n\r\n"
            connection.send(content: Data(httpRequest.utf8), completion: .contentProcessed { error in
                if let error = error {
                    finish(.failure(error))
                } else {
                    receiveLoop()
                }
            })
        case .failed(let error):
            finish(.failure(error))
        case .waiting:
            // Not a failure. This fires while there's no usable cellular path
            // yet, e.g. the radio is still waking up because WiFi is active.
            // Keep waiting; the timeout below bounds how long.
            break
        default:
            break
        }
    }

    connection.start(queue: queue)

    // Hard timeout: covers a radio that never comes up (stuck in .waiting)
    // as well as a server that never finishes responding.
    queue.asyncAfter(deadline: .now() + timeout) {
        finish(.failure(CellularRequestError.timedOut))
    }
}
```

Then, for example, to get the device's cellular IPv6 for `deviceIp`:

```swift title="Usage Example"
forceCellularRequest(urlString: "https://your-api.example.com/device-ip") { result in
    switch result {
    case .success(let ipv6):
        // Pass this to your server as deviceIp for
        // GET /1-click/verifications/channels
        break
    case .failure:
        // No cellular available (or request failed): autofill won't be
        // available, so fall back to the SMS channel.
        break
    }
}
```

:::note
The completion handler fires on a private queue, not the main thread. Hop to the main thread before touching UI. No special permissions or entitlements are needed on iOS.
:::


## Android

On Android, use `ConnectivityManager.requestNetwork()` with a `NetworkRequest` that specifies `NetworkCapabilities.TRANSPORT_CELLULAR`. The system hands you a `Network` object that's guaranteed to be cellular (waking the radio if needed). Binding your HTTP client to that `Network` is what actually forces the traffic over cellular; otherwise Android routes requests over WiFi whenever it's connected.

Two details are easy to get wrong:

1. **DNS must be bound separately.** The `socketFactory` only covers the TCP sockets; without OkHttp's `dns` hook, lookups still resolve over WiFi.
2. **Unregister the callback when you're done.** An active `requestNetwork` registration holds the cellular radio up indefinitely, and Android caps an app at 100 outstanding registrations.

```kotlin title="CellularRequest.kt"
class NoCellularAvailableException : IOException("No cellular network available")

fun forceCellularRequest(url: String, connectivityManager: ConnectivityManager, timeoutMs: Int, callback: (Result<String>) -> Unit) {
    val request = NetworkRequest.Builder()
        .addTransportType(NetworkCapabilities.TRANSPORT_CELLULAR)
        .build()

    connectivityManager.requestNetwork(request, object : ConnectivityManager.NetworkCallback() {
        // onAvailable can fire again if cellular drops and comes back;
        // only run the request once.
        private val fired = AtomicBoolean(false)

        override fun onAvailable(network: Network) {
            if (!fired.compareAndSet(false, true)) return
            val networkCallback = this

            val client = OkHttpClient.Builder()
                .socketFactory(network.socketFactory)
                // socketFactory only covers the TCP sockets. Without this,
                // DNS still resolves over the default network (WiFi).
                .dns(object : Dns {
                    override fun lookup(hostname: String): List<InetAddress> =
                        network.getAllByName(hostname).toList()
                })
                .build()

            val httpRequest = Request.Builder().url(url).get().build()

            client.newCall(httpRequest).enqueue(object : Callback {
                override fun onFailure(call: Call, e: IOException) {
                    connectivityManager.unregisterNetworkCallback(networkCallback)
                    callback(Result.failure(e))
                }
                override fun onResponse(call: Call, response: Response) {
                    // Read the body before releasing the network, then
                    // unregister so the OS can drop the cellular radio again.
                    val result = response.use {
                        if (it.isSuccessful) {
                            Result.success(it.body?.string().orEmpty())
                        } else {
                            Result.failure(IOException("HTTP ${it.code}"))
                        }
                    }
                    connectivityManager.unregisterNetworkCallback(networkCallback)
                    callback(result)
                }
            })
        }
        override fun onUnavailable() {
            // The system releases the request automatically after onUnavailable,
            // so no unregister is needed on this path.
            callback(Result.failure(NoCellularAvailableException()))
        }
    }, timeoutMs)
}
```

Usage looks the same as iOS: request your IP echo endpoint, pass the result to your server as `deviceIp`, and fall back to the SMS channel on failure.

:::important Permissions and API Level
- `requestNetwork()` requires the `CHANGE_NETWORK_STATE` permission. Also declare `ACCESS_NETWORK_STATE` and `INTERNET`. All three are normal install-time permissions with no runtime prompt.
- The `requestNetwork` overload that takes a timeout, and `onUnavailable()`, require **API 26+**.
:::


## Cross-Platform Frameworks

If your app is built on a cross-platform framework, the approach is the same: implement the two native functions above, then expose them through your framework's native bridge.

```swift title="CellularRequestModule.swift"
@objc(CellularRequestModule)
class CellularRequestModule: NSObject {
    @objc static func requiresMainQueueSetup() -> Bool { false }

    @objc
    func forceCellularRequest(_ urlString: String,
                               resolver: @escaping RCTPromiseResolveBlock,
                               rejecter: @escaping RCTPromiseRejectBlock) {
        forceCellularRequest(urlString: urlString) { result in
            switch result {
            case .success(let body):
                resolver(body)
            case .failure(let error):
                rejecter("CELLULAR_REQUEST_FAILED", error.localizedDescription, error)
            }
        }
    }
}
```

```kotlin title="CellularRequestModule.kt"
class CellularRequestModule(reactContext: ReactApplicationContext) :
    ReactContextBaseJavaModule(reactContext) {

    override fun getName() = "CellularRequestModule"

    @ReactMethod
    fun forceCellularRequest(url: String, promise: Promise) {

            .getSystemService(Context.CONNECTIVITY_SERVICE) as ConnectivityManager

        forceCellularRequest(url, connectivityManager, timeoutMs = 10_000) { result ->
            result.fold(
                onSuccess = { promise.resolve(it) },
                onFailure = { promise.reject("CELLULAR_REQUEST_FAILED", it.message, it) }
```

<TabItem value="flutter" label="Flutter">

Expose the native functions through a method channel.

```dart title="cellular_request.dart"
import 'package:flutter/services.dart';

class CellularRequest {
  static const _channel = MethodChannel('cellular_request');

  static Future<String> forceCellularRequest(String url) async {
    try {
      final result = await _channel.invokeMethod<String>('forceCellularRequest', {'url': url});
      return result ?? '';
    } on PlatformException catch (e) {
      throw Exception('${e.code}: ${e.message}');
    }
  }
}
```

iOS handler (in `AppDelegate.swift`):

```swift title="AppDelegate.swift (handler excerpt)"
channel.setMethodCallHandler { call, result in
  guard call.method == "forceCellularRequest" else {
    result(FlutterMethodNotImplemented)
    return
  }
  guard let args = call.arguments as? [String: Any],
        let urlString = args["url"] as? String else {
    result(FlutterError(code: "BAD_ARGS", message: "Missing 'url'", details: nil))
    return
  }
  forceCellularRequest(urlString: urlString) { outcome in
    // The completion fires on the request's private queue; Flutter
    // results must be delivered on the main thread.
    DispatchQueue.main.async {
      switch outcome {
      case .success(let body):
        result(body)
      case .failure(let error):
        result(FlutterError(code: "CELLULAR_REQUEST_FAILED", message: error.localizedDescription, details: nil))
      }
    }
  }
}
```

Android handler (in `MainActivity.kt`):

```kotlin title="MainActivity.kt (handler excerpt)"
val mainHandler = Handler(Looper.getMainLooper())

MethodChannel(flutterEngine.dartExecutor.binaryMessenger, "cellular_request")
    .setMethodCallHandler { call, result ->
        if (call.method != "forceCellularRequest") {
            result.notImplemented()
            return@setMethodCallHandler
        }
        val url = call.argument<String>("url")
        if (url == null) {
            result.error("BAD_ARGS", "Missing 'url'", null)
            return@setMethodCallHandler
        }
        forceCellularRequest(url, connectivityManager, timeoutMs = 10_000) { outcome ->
            // The callback arrives on a background (OkHttp) thread;
            // MethodChannel results must be delivered on the main thread.
            mainHandler.post {
                outcome.fold(
                    onSuccess = { result.success(it) },
                    onFailure = { result.error("CELLULAR_REQUEST_FAILED", it.message, null) }
                )
            }
        }
    }
```

<TabItem value="other" label="Other Frameworks">

The same pattern applies to any framework with a native bridge:

- **.NET MAUI / Xamarin**: define a shared interface (for example `Task<string> ForceCellularRequestAsync(string url)`) and implement it per platform. .NET for Android exposes `Android.Net.ConnectivityManager` directly, and .NET for iOS exposes `Network.framework` under the `Network` namespace.
- **Kotlin Multiplatform**: declare `expect suspend fun forceCellularRequest(url: String): Result<String>` and provide `actual` implementations in the iOS and Android source sets.
- **Capacitor / Cordova**: build a native plugin backed by the same two implementations. Note that this only covers requests the *plugin* makes natively. A `fetch()` running in the page's own JavaScript inside the WebView can't be forced onto cellular on iOS, so route cellular-dependent requests through the plugin.

## Good to Know

- **Handle "no cellular" as a normal outcome, not an edge case.** Airplane mode, no SIM, no data plan, and data saver all mean the request will fail or time out. When that happens, fall back to the [SMS channel](/1-click-verify/guides/api-integration#2-if-necessary-use-sms-channel).
- **Always use a timeout.** Don't let a request hang waiting for a cellular network that will never come up.
- **Waking the radio takes a moment.** If cellular was idle (because WiFi was handling everything), forcing a request over it wakes the radio, which adds a brief delay and uses a small amount of metered data.
- **Dual SIM / eSIM**: these APIs give you "a cellular network," not a specific SIM line. Neither platform lets you target one SIM over another this way.
- **Test on real devices with real carriers.** Simulators and emulators don't model network interface behavior reliably.
